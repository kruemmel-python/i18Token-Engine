1. Die Vision: Spirit & Body

In der modernen Software-Entwicklung trennen wir den Geist (die Logik/Python) vom Koerper (die Daten/Texte). Die i18nToken Engine ist das Bindeglied.

Warum nutzen wir Hexadezimal-Tokens statt normaler Woerter?

  • Stabilitaet: Der Code get_text("0xABC") muss nie geaendert werden, egal ob der Text dahinter "Hallo" oder "Willkommen zurueck, Reisender" heisst.
  • Sicherheit: Uebersetzer koennen keine Bugs im Code verursachen.

2. Das Abschlussprojekt: "Token Quest"

Wir bauen ein Text-Adventure. Die gesamte Welt wird in einer Textdatei definiert. Hier ist der Aufbau der world.txt:

world.txtUTF-8 Encoding
# Raumbeschreibungen
100(Room_Start): Du stehst in einer dunklen Bibliothek. @200
101(Room_Cellar): Ein feuchter Keller. Du siehst %0.
200(Detail_Books): Ueberall stehen Regale voller Python-Buecher.

# Interaktionen
300(Action_Take): Du hast %0 aufgehoben.
400(Error_Move): Hier geht es nicht weiter!

Die Spiel-Logik in Python

In Python steuern wir nun den Fluss. Wir setzen Platzhalter ein, um das Spiel lebendig zu machen.

adventure.py
# Beispiel fuer die Spiel-Schleife
current_room = "100"
inventory = "Einen goldenen Token"

# Raum beschreiben
print(engine.translate(current_room))

# Aktion ausfuehren (mit Platzhalter)
print(engine.translate("300", [inventory]))

3. Produktions-Workflow

  1. Katalog bauen: Tokenliste anlegen und Texte einfuegen.
  2. Logik verbinden: Python ruft Tokens statt Klartexten auf.
  3. QA-Check: check_report() ausfuehren und Logs beheben.
  4. Playtest: Story laufen lassen, Platzhalter testen.
  5. Release: Katalog einfrieren, neue Sprache anlegen.

Goldene Regel: Jede Aenderung am Katalog endet mit einem QA-Report ohne ERROR.

4. Release & Pluralformen

Token sind jetzt mehrsprachige Bausteine mit Varianten wie {one} oder {other}. Rufe engine.translate_plural(token, count, args) aus deiner Logik auf, um automatisch die passende Form auszuliefern.

Plural-Call
print(engine.translate_plural("f1e2d3", item_count, [item_name]))

Vor dem Deployment exportierst du den Katalog via python i18n_crypt.py --strict locales/de.txt releases/de.i18n. Die Engine erkennt das Magic-Header-Format und lädt den Binärkatalog in Sekunden.

Release-Check: Bestätige, dass dein releases/*.i18n-Artefakt eine gültige Checksum hat und deine Pluralvarianten enthalten sind.

5. Missions-Board

Challenge A: QA-Meister

Die Engine meldet:
ERROR CYCLE: 101 -> 102 -> 101

Was ist passiert? Korrigiere die Logik im Kopf!

Challenge B: Release-Workflow

Wie schuetzt du deine world.txt vor Cheatern und stellst ein sauberes Release sicher?

Nutze das neue CLI: python i18n_crypt.py --strict world.txt releases/world.i18n, damit die Engine das binäre Artefakt lädt.

Challenge C: Platzhalter-Test

Fuege eine neue Aktion mit %0 und %1 ein und teste mit zwei Items.

Ziel: Keine WARN GAP im QA-Report.

Challenge D: Lost Token

Simuliere einen fehlenden Token im Katalog und erklaere, warum strict=True sinnvoll ist.

6. Review & Freeze

Die finale Tour: Contract, Tests, Meta-Infos und Overflow melden ist jetzt State-of-the-art. Nur glatte Runs erhalten die finale Zertifizierung.

  • Contract Guard: `CONTRACT.md` dokumentiert ABI, Binary-Format, Language Spec und the Freeze Plan.
  • Golden Tests: `tests/run_tests.py` prüft Meta (Locale/Fallback/Note/Plural), QA, Escape-Regeln und `RESULT_TOO_LARGE`.
  • Release Protocol: Contract-Review → Tests → Binary-Export (`i18n_crypt.py`) → Deployment.

Erst danach: Certificate, Release, und das Training ist wirklich zertifiziert.
